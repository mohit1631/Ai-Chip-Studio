How a bug is supposed to be caught — and what happens if it isn't

Chip design splits into two eras: pre-silicon, while the design still exists only as code and simulation, and post-silicon, after the design has been fabricated into real, physical, unchangeable silicon. The goal of every verification team is to push as many bugs as possible to the left of that line, because the cost of fixing them rises by orders of magnitude once you cross it.

STAGE 01 · PRE-SILICON

RTL Design & Static Lint

Engineers write the chip's logic in Verilog/VHDL. Lint tools (Synopsys SpyGlass, Cadence HAL, or simpler open tools) scan the code for structural red flags — latch inference, multi-driven nets, blocking/non-blocking misuse — before a single simulation even runs. This is the stage this site's checker tool operates at.

STAGE 02 · PRE-SILICON

Simulation

The RTL is run against millions of test vectors in a software simulator. Every clock cycle of intended behavior is checked against actual behavior. This is where the bulk of functional bugs are found — but simulation can only test the cases someone thought to write.

STAGE 03 · PRE-SILICON

Formal Verification

Mathematical proof engines check that the design satisfies its specification for all possible inputs, not just the ones tested. Used heavily for control logic, cache coherence, and security-critical paths — places where an untested corner case is unacceptable.

STAGE 04 · PRE-SILICON

Emulation & FPGA Prototyping

The design is mapped onto large FPGA arrays or hardware emulators so it can run at near-real speed — enough to boot an OS or run real software/games on a CPU/GPU design before it's ever fabricated. Bugs that only appear after billions of cycles surface here.

STAGE 05 · PRE-SILICON

Tape-out

The final, verified design is converted into photomask data and sent to a fab (TSMC, Samsung, Intel Foundry) to be physically manufactured. This is the point of no easy return — once silicon is cut, the logic is frozen in physical transistors.

↓ a bug surviving past this line is now physically baked into the chip ↓
STAGE 06 · POST-SILICON

Bring-Up & Validation

The first real chips come back from the fab and engineers power them on in a lab for the first time, running diagnostic and real-world workloads to see if behavior matches simulation. This is where bugs that escaped every pre-silicon check finally show up.

STAGE 07 · POST-SILICON

Errata & Workarounds

Confirmed bugs get logged in a public errata document. If possible, the fix ships as a microcode or firmware/driver patch — software papering over a hardware mistake — rather than touching the silicon itself.

STAGE 08 · POST-SILICON

Re-spin (Last Resort)

If a bug can't be patched around — it's too severe or too deep in the logic — the design must be corrected and sent back to the fab for new masks and a new manufacturing run. This costs millions of dollars and months of schedule, which is the entire reason stages 1–5 exist.

When a bug crosses from pre-silicon into real chips

Re-spin / Recall

Intel Pentium FDIV Bug

1994

A floating-point division lookup table had five incorrect entries out of over a thousand, so certain divisions returned a slightly wrong answer. The error was subtle enough that it went unnoticed for roughly a year before a mathematician found it.

Intel initially resisted a full recall, which caused public backlash and pressure from partners like IBM, before eventually recalling all affected chips at a cost of roughly $475 million — one of the clearest examples of how expensive a single bad data value can be once it's already in silicon.

Microcode Patch

Spectre & Meltdown-class Speculative Execution Bugs

2018 – ongoing

These are architectural side-effects of speculative execution — a performance feature — that let attackers read memory they shouldn't have access to. Because the issue is baked into how the processor's pipeline behaves, it can't be fully "fixed" with a simple patch.

Affected vendors ship microcode and OS-level mitigations instead of recalling hardware, often trading some performance for safety. Variants of this same class of bug have continued to be discovered in newer CPU generations years later.

Errata / Workaround

Hyperthreading & AVX-Related Errata

Various generations

Modern CPU errata sheets regularly list dozens to over a hundred known issues per generation — things like specific instruction sequences causing unpredictable behavior, or rare machine-check exceptions under heavy load.

Most are resolved with microcode updates pushed through OS or BIOS updates rather than hardware changes, but a few have been serious enough that vendors or distros recommended disabling a whole feature (like hyperthreading) until a fix shipped.

Voltage / Degradation

Intel 13th/14th Gen Instability Issue

2024

Some desktop CPUs were found to request more voltage than intended under certain conditions over time, which could permanently degrade the silicon and cause crashes in affected systems.

The fix required both a microcode update to correct the voltage request behavior going forward, and in some cases warranty replacement for chips already degraded — illustrating that even issues found post-launch don't always have a clean software-only fix.

The common thread: every one of these passed through real verification teams and still escaped to silicon. That's not a sign verification doesn't work — modern chips have billions of transistors and effectively infinite input combinations, so 100% coverage pre-silicon is not achievable. The goal of tools like the static + AI checker on this site is to catch the bugs that are catchable early, so the remaining post-silicon risk is as small and as cheap as possible.
